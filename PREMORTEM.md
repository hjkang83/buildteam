# 말하는 비서실 Pre-mortem
> 4/11 팀원들과의 Pre-mortem 논의를 기반으로 정리
> "6개월 후, 이 프로젝트가 망했다고 가정하자. 왜 망했을까?"

---

## Pre-mortem이란?

프로젝트 시작 전에 "이미 실패했다"고 가정하고, 실패 원인을 미리 상상하여 사전에 대비하는 분석 방법이다. 낙관 편향을 깨고, 현실적인 리스크를 직시하기 위해 수행한다.

---

## 실패 영역 1: 기술적 한계 및 구현 실패

> 팀 논의에서 가장 많은 우려가 나온 영역

---

### 시나리오 1-1: 데이터 크롤링 실패

**상상된 실패:**
> "부동산 명저 20권과 실시간 매물 데이터를 학습시키려 했는데, 크롤링이 제대로 안 돼서 에이전트가 학습할 소스 자체가 부족했다. 멘토 에이전트가 명저를 인용해야 하는데, 정작 인용할 지식이 없었다."

**사전 대응:**

| 대응 | 구체적 행동 | 시기 |
|------|------------|------|
| 데이터 수집 우선 착수 | 코드 개발보다 명저 텍스트 확보를 1순위로 진행 | 1주차 |
| 수동 입력 허용 | 크롤링 실패 시, 핵심 구절을 수동으로 텍스트 파일화하는 플랜B | 2주차 |
| MVP 범위 축소 | 20권 전부가 아닌 핵심 3권만으로도 MVP 가능하게 설계 | 3주차 판단 |
| RAG 품질 테스트 | 데이터 확보 즉시 "이 질문에 이 답이 나오는가?" 검증 | 상시 |

---

### 시나리오 1-2: 지연 시간(Latency)이 대화를 죽인다

**상상된 실패:**
> "STT → LLM → TTS를 거치다 보니 사용자가 말한 후 응답까지 5초 이상 걸렸다. 쿠션 멘트를 넣어도 대화 리듬이 끊어져서 실제 사람과 회의하는 느낌이 전혀 나지 않았다. 사용성이 최악이었다."

**사전 대응:**

| 대응 | 구체적 행동 | 시기 |
|------|------------|------|
| 파이프라인 병렬 처리 | STT 스트리밍 + LLM 스트리밍 + TTS 스트리밍을 연결 (Pipecat 활용) | 4주차 |
| 쿠션 멘트 설계 | "네", "음...", "잠시만요" 등 0.5초 내 선발화 멘트 라이브러리 구축 | 5주차 |
| LLM 분기 전략 | 간단한 추임새는 GPT-4o-mini(빠름), 심층 분석만 Opus(느림) | 6주차 |
| 체감 지연 측정 | "말 끝 → 응답 시작"까지 2초 이내를 KPI로 설정하고 반복 측정 | 7주차~ |

---

### 시나리오 1-3: 에이전트 역할이 뒤죽박죽

**상상된 실패:**
> "에이전트의 역할을 정교하게 짜지 못해서 실무 에이전트가 멘토처럼 말하고, 레드팀이 찬성만 했다. 페르소나가 구분이 안 되니 3명이 있어도 1명과 말하는 것 같았다. 전체 시스템이 잘 안 돌아갔다."

**사전 대응:**

| 대응 | 구체적 행동 | 시기 |
|------|------------|------|
| 페르소나 명세서 작성 | 각 에이전트의 말투, 금지 행동, 필수 행동을 문서화 | 3주차 |
| 대조 테스트 | 같은 질문에 3명의 응답을 나란히 놓고 "구분이 되는가?" 검증 | 6주차 |
| 레드팀 강제 규칙 | 레드팀 프롬프트에 "반드시 1개 이상의 반론을 제시할 것" 명시 | 4주차 |
| 말투 샘플 확보 | PPT의 유저 씬 대화를 기준 응답(Gold Standard)으로 설정 | 3주차 |

---

### 시나리오 1-4: 액션으로 이어지지 않는다

**상상된 실패:**
> "역할만 부여됐을 뿐, 실제 구체적인 액션으로 이어지는 기능 구현에 실패했다. 대화는 했는데 회의록 자동 생성도 안 되고, 넥스트 액션 플랜도 안 나오고, 결국 '말만 하는' 앱이 됐다."

**사전 대응:**

| 대응 | 구체적 행동 | 시기 |
|------|------------|------|
| 서기 에이전트 우선 개발 | 대화 기록 → 요약 → 액션 도출 파이프라인을 핵심 기능으로 | 5주차 |
| 출력 포맷 고정 | 회의록 마크다운 템플릿 사전 정의 (결정사항 / 보류사항 / 넥스트액션) | 4주차 |
| 옵시디언 연동 테스트 | 마크다운 자동 저장이 실제로 작동하는지 조기 검증 | 7주차 |
| MVP 최소 기준 | "회의 끝나면 요약본이 나온다"가 안 되면 MVP로 인정 안 함 | 상시 |

---

### 시나리오 1-5: 음성 프롬프트 처리 미흡

**상상된 실패:**
> "말로 하는 프롬프트가 정제되지 않아 AI가 의도를 파악하지 못했다. '이거 좀 알아봐줘'라고 했는데 '이거'가 뭔지 몰라서 엉뚱한 답을 했다. 텍스트와 달리 음성은 모호함이 훨씬 많았다."

**사전 대응:**

| 대응 | 구체적 행동 | 시기 |
|------|------------|------|
| 모호성 감지 로직 | "이거", "그거", "아까 그" 같은 지시어 감지 시 역질문 트리거 | 5주차 |
| 대화 맥락 유지 | 직전 3~5턴의 대화를 LLM 컨텍스트에 항상 포함 | 4주차 |
| 역질문 프롬프트 | "사용자의 의도가 불명확하면 추정하지 말고 반드시 되물어라" 명시 | 4주차 |
| 실제 구어체 테스트 | 정제된 문장이 아닌, 실제 말하듯 흐릿한 문장으로 테스트 | 7주차~ |

---

### 시나리오 1-6: 우리 실력이 부족하다

**상상된 실패:**
> "구상은 거창했는데, 우리 실력이 부족해서 끝내 구현하지 못했다. STT-LLM-TTS 파이프라인을 연결하는 것부터 막혔고, Pipecat 문서를 읽어도 이해가 안 됐다."

**사전 대응:**

| 대응 | 구체적 행동 | 시기 |
|------|------------|------|
| 최소 파이프라인 먼저 | STT → LLM → TTS 단순 연결부터 성공시키고 기능을 덧붙이기 | 3주차 |
| Claude Code 활용 | 코드 작성은 Claude Code에 위임, 팀은 설계와 검증에 집중 | 상시 |
| 튜토리얼 기반 학습 | Pipecat 공식 예제를 복사 → 수정하는 방식으로 진입 | 2주차 |
| 기능 우선순위 | 전부 못 만들면 "1대1 음성 대화 + 회의록"만이라도 완성 | 8주차 판단 |

## 실패 영역 2: 팀워크 및 실행력

> 기술보다 더 무서운 실패 원인

---

### 시나리오 2-1: 누군가 하겠지 증후군

**상상된 실패:**
> "서로 바쁘다는 핑계로 '누군가 하겠지' 하며 미루다가 시간만 다 보냈다. 마감 2주 전에야 정신 차렸지만 이미 늦었다."

**사전 대응:**

| 대응 | 구체적 행동 | 시기 |
|------|------------|------|
| 주간 마일스톤 | 매주 "이번 주에 이것만은 완료" 목표를 정하고 공유 | 매주 |
| 담당자 명시 | 모든 작업에 이름을 붙임. "우리가 한다" = 아무도 안 한다 | 상시 |
| 주간 모임 필수 | 할 일이 없어도 모여서 진행 상황 공유 | 매주 |
| 조기 경보 | 2주 연속 진전 없으면 범위 축소를 논의 | 상시 |

---

### 시나리오 2-2: 겉으로만 합의, 속으로는 딴생각

**상상된 실패:**
> "앞에서는 다들 좋다고 했는데, 속으로는 다 딴생각 하느라 동력이 없었다. 진심으로 이 프로젝트를 하고 싶은 사람이 없었다."

**사전 대응:**

| 대응 | 구체적 행동 | 시기 |
|------|------------|------|
| 솔직한 온도 체크 | 정기적으로 "지금 이 프로젝트 몰입도 1~10점?" 익명 조사 | 격주 |
| 개인 동기 연결 | 각자 "이 프로젝트에서 내가 얻고 싶은 것" 명확화 | 1주차 |
| 빠른 성공 경험 | 2주 내 "일단 돌아가는 것"을 만들어서 팀에 활력 부여 | 3주차 |
| 갈등은 투표로 | 의견 대립 시 카카오톡 익명 투표. 결과에 전원 승복 | 상시 |

---

### 시나리오 2-3: 동상이몽

**상상된 실패:**
> "디테일에 대한 서로의 가정과 정의가 달라서 동상이몽인 결과물이 나왔다. '에이전트가 대화한다'의 의미를 어떤 팀원은 '짧은 응답', 다른 팀원은 '긴 설명'으로 이해하고 있었다."

**사전 대응:**

| 대응 | 구체적 행동 | 시기 |
|------|------------|------|
| 용어 사전 | "에이전트", "티키타카", "쿠션 멘트" 등 핵심 용어의 정의를 문서화 | 1주차 |
| 유저 씬 공유 | PPT의 부동산 대화 예시를 "이게 우리가 만들 것"의 기준으로 삼기 | 1주차 |
| MANIFEST 참조 | 핵심 가치 문서를 팀 전원이 읽고 동의한 뒤 시작 | 1주차 |
| 데모 기반 소통 | 추상적 논의 대신 "이걸 실행하면 이렇게 나와야 해"를 기준으로 대화 | 상시 |

## 핵심 교훈

이 Pre-mortem에서 반복적으로 드러나는 패턴 2가지:

### 1. "기술의 적은 지연 시간이다"
STT → LLM → TTS 파이프라인의 총 지연 시간이 이 서비스의 생사를 결정한다. 아무리 똑똑한 에이전트라도 5초 동안 침묵하면 사용자는 떠난다. **Latency가 곧 UX다.**

### 2. "팀의 적은 암묵적 합의다"
"다들 좋다고 했으니 됐겠지"가 가장 위험하다. 말하지 않은 가정, 확인하지 않은 정의, 미뤄둔 결정이 쌓이면 동상이몽 결과물이 나온다. **명시적 합의만이 진짜 합의다.**
