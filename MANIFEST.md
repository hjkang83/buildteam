# 말하는 비서실 Manifest
> 우리가 왜 이걸 만드는지, 어떤 기준으로 결정하는지에 대한 약속

---

## 한 줄 정의

**목소리만으로 전문가 팀과 회의하듯 일할 수 있는 AI 비서 시스템**

---

## Core Values (핵심 가치)

### 1. 리더의 인터페이스 (Leader's Interface)

텍스트를 치는 건 비서에게 보고서를 쓰라는 것이지, 비서에게 말을 거는 것이 아니다. 회장님이 비서진에게 하듯 구두로 핵심을 찌르는 지휘가 가능해야 한다. 프롬프트를 짜는 것은 또 다른 종류의 노동이다. 우리는 그 노동을 "대화"로 대체한다.

**실천 기준:**
- 사용자가 키보드를 한 번도 치지 않아도 회의가 완결되어야 한다
- "프롬프트를 잘 짜야 좋은 답이 나옵니다"는 우리 서비스에서 실패를 의미한다
- 막연한 말 한마디에도 에이전트가 역질문으로 의도를 파악해야 한다

### 2. 입체적 전문가 집단 (Multi-Perspective Intelligence)

한 명의 AI가 "네, 좋은 생각입니다"만 반복하면 인간 비서보다 못하다. 찬성, 반대, 조언자로 역할이 나뉜 에이전트들이 서로 다른 관점을 제시하여 정보 편향 없는 객관적 판단을 지원한다. 리더에게 가장 위험한 것은 예스맨으로 둘러싸이는 것이다.

**실천 기준:**
- 최소 3가지 페르소나(실무/레드팀/멘토)가 확연히 다른 관점을 제시할 것
- 레드팀 에이전트는 반드시 반론을 제기하도록 설계할 것
- 사용자의 의견에 무조건 동조하는 응답이 나오면 프롬프트를 수정할 것

### 3. 마스터급 멘토링 (Deep Wisdom, Not Just Data)

단순히 데이터를 읊는 건 검색 엔진이면 충분하다. 각 분야 명저와 철학을 학습한 멘토 에이전트가 사용자의 마인드셋까지 가이드한다. "수익률이 몇 %입니다"가 아니라 "지금 진짜 필요한 게 타이틀인지, 현금 흐름인지 본질을 짚어보시죠"라고 말할 수 있어야 한다.

**실천 기준:**
- 멘토 에이전트는 명저 기반 RAG 데이터를 활용할 것
- 숫자만 나열하는 응답은 실무 에이전트의 몫. 멘토는 "왜?"에 답할 것
- 사용자가 "아, 그런 관점도 있구나"라고 느낄 수 있어야 한다

### 4. 끊김 없는 경영 연속성 (Seamless Continuity)

대화로 끝나면 그냥 수다다. 회의록 작성, 넥스트 액션 도출, 다음 회의 아젠다 세팅까지 시스템이 능동적으로 수행한다. 오늘 회의에서 나온 결정이 내일 회의의 시작점이 되는 선순환 구조를 만든다.

**실천 기준:**
- 회의 종료 시 반드시 핵심 결정사항 + 넥스트 액션이 자동 생성될 것
- 다음 회의 시작 시 이전 회의 맥락이 RAG로 자동 로드될 것
- 사용자가 "지난번에 뭐라고 했더라?"라고 물으면 바로 답할 수 있을 것

## Design Principles (설계 원칙)

### 1. 사람처럼 대화한다 (Human-like Conversation)

에이전트는 기계가 아니라 회의 참석자다. 추임새("아, 그렇군요"), 쿠션 멘트("잠시만요, 데이터 확인 중입니다"), 역질문("수익률이라 하시면 레버리지 기준인가요?")을 구사한다. 답변은 최대 2문장. 긴 독백은 회의가 아니라 강의다.

### 2. 지연을 숨기지 말고, 자연스럽게 메운다 (Latency Masking)

AI 추론에 시간이 걸리는 건 피할 수 없다. 그 시간을 어색한 침묵으로 두지 않고, "지금 계산기 돌리고 있습니다" 같은 쿠션 멘트로 대화 리듬을 유지한다. 사용자가 기다리고 있다고 느끼면 안 된다.

### 3. 사용자가 말하면 즉시 멈춘다 (Instant Cut-off)

사용자가 개입하면 0.1초 내에 에이전트 발화를 중단하고 경청 모드로 전환한다. AI가 사용자의 말을 끊는 것은 절대 용납되지 않는다.

### 4. 예쁜 쓰레기보다 투박한 명품 (Function over Form)

iOS/Android 앱의 화려함보다 웹 기반으로 빠르게 구축한다. 초기에는 대화의 품질(프롬프트 엔지니어링, 에이전트 간 인터랙션 로직)에 자원을 집중한다. UI는 기능이 안정된 후에 다듬는다.

---

## 이 Manifest의 사용법

- 기능 추가를 고민할 때 → 핵심 가치에 부합하는가?
- "이거 텍스트 입력도 지원하자" → 리더의 인터페이스 원칙에 위배되지 않는가?
- "에이전트가 너무 길게 말한다" → 설계 원칙 1번 위반
- 방향을 잃었을 때 → 맨 위 "한 줄 정의"로 돌아온다
